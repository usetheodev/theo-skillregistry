# Theo Skill Registry — Roadmap

> Generated by `/roadmap-init` on 2026-06-22 from slug `skill-registry`.
> This is a **macro** roadmap. Decomposition into tasks is the job of `/to-plan` per milestone.

## Vision

O Theo Skill Registry é o **registro remoto de skills que o Theokit ainda não tem** e o
substituto superior do Google Cloud Skill Registry: skills versionadas no formato `SKILL.md`
**nativo do Theokit**, descobertas por **busca híbrida transparente** (keyword + vetor, com
score explícito — não a busca opaca do Google), com a DX (CLI de validação local) e a
governança (RBAC por skill, observabilidade por skill, webhook de operações) que o Google
não entrega. Vence quando um agente Theokit encontra a skill certa para a intenção do
usuário em milissegundos.

## Problem

**Root problem:** Agentes do ecossistema Theo precisam de skills versionadas, governadas e
descobríveis por intenção. O Theokit hoje só descobre skills do **filesystem local**
(`.theokit/skills/<name>/SKILL.md`), sem registro remoto e sem busca semântica — o LLM vê
uma lista plana. O Google Skill Registry resolve o registro remoto, mas com busca opaca
(sem score/modelo), LRO só por polling, IAM all-or-nothing por projeto, sem observabilidade
por skill e sem CLI de dev local.

**Whose pain:** Times do Theo que constroem agentes Theokit e precisam publicar, versionar
e descobrir skills com confiança — hoje sem fonte única de verdade nem descoberta por
intenção.

## Users

- **Primary:** Skill Author + Agent Builder (engenheiros internos do Theo), e o **runtime do
  Theokit** (`SkillsManager`) como consumidor de máquina (interno, múltiplos times).
- **Secondary:** Operador/SRE que audita e governa o catálogo de skills.

## Scope

### In scope (V1 — required to call this project alive)

- **Paridade de gestão, formato Theokit:** CRUD de skills + revisões imutáveis + LRO (com
  webhook, não só polling) sobre o formato `SKILL.md` nativo do Theokit
  (`name/version/requires/description/allowed-tools/category/dependencies`).
- **Descoberta superior à do Google:** busca híbrida (keyword + vetor via pgvector) com
  reranking e **score transparente** — Recall@5 ≥ 0.85 e retrieve p95 < 200ms.
- **DX + governança:** CLI de validação local pré-upload (estrutura, limites de payload,
  secret scan) + RBAC granular por skill + integração remota com o Theokit (o provider que
  ele ainda não tem), validada por **dogfood real**.

### Explicitly out of scope

- Execução/runtime de skills — *why excluded:* é responsabilidade do **Theokit** (substituto
  do ADK da casa), não do registry. Nós armazenamos e descobrimos; ele executa.
- Marketplace público / skills da comunidade — *why excluded:* V1 é interno ao Theo;
  catálogo público adiciona moderação, segurança e curadoria fora do escopo de V1.
- Composição/dependências entre skills (resolução transitiva + conflito) — *why excluded:*
  acoplamento prematuro; só justifica-se com casos concretos depois do V1.
- Compliance pesado (CMEK / VPC-SC / HIPAA) — *why excluded:* diferencial forte mas caro;
  pós-V1, com ADR próprio.

> Items in this list are off-limits for V1. To reconsider, write a new roadmap revision — do not silently expand scope.

## Constraints

| Category | Constraint |
|---|---|

| Compliance / legal | CMEK/VPC-SC/HIPAA fora do V1. Peers de estudo sem licença permissiva (`anthropic-skills`, `openskills`) são **study-only** — não copiar código. |
| Deadline | Nenhum prazo fixo declarado. |
| Team | Pequeno, interno ao Theo. |
| Runtime target | Retrieve p95 < 200ms; formato e API de descoberta devem casar com o `SkillsManager` do Theokit. |
| Budget (infra, tooling) | Infra mínima: um Postgres + pgvector. Sem dependências extras de infra. |

## Success criteria

**V1 ship criterion (measurable):** Recall@5 ≥ 0.85 na busca de skills **E** retrieve
p95 < 200ms **E** dogfood real — um agente Theokit consumindo o registry remoto (provider
HTTP) em uso interno de verdade, não sintético.

> Example shape: "100 req/s sustained P95 < 200ms across 3 providers, with zero retries hidden from caller", NOT "feels fast".

**North-star metric (tracked post-launch):** **Time-to-relevant-skill** — latência +
precisão combinadas: quão rápido e certo um agente Theokit encontra a skill correta para a
intenção do usuário.

> The number that, when it moves, the team knows the project is winning. Differs from ship criterion — ship is "alive", north-star is "winning".

## Milestones

> Each milestone has a checkbox in its header. Flip `[ ]` → `[x]` as you complete it. Status lives in this document; no external tracker required.

### M0 — [x] Walking skeleton

**Objective:** Provar a arquitetura fim-a-fim na fatia mais fina: criar uma skill (zip
`SKILL.md`) via LRO e recuperá-la, ponta a ponta, na stack da casa.

**Definition of done (all must hold):**

- [x] Monorepo pnpm (`packages/core` + `packages/api`) compila em TS strict; Hono sobe com `/v1/health`.
- [x] `POST /v1/skills` enfileira uma operação via pg-boss e `GET /v1/operations/{id}` reporta `done`; `GET /v1/skills/{id}` retorna a skill persistida (Postgres + Drizzle, migrations aplicadas).
- [x] Teste E2E: criar → aguardar operação → obter a skill, verde no CI.

**Dependencies:** none (this is the foundation).

**Top risks:**

1. pg-boss e Hono mal integrados ao monorepo (graceful shutdown ordenado: server → queue → pool).
2. Escopo do skeleton inflar para além de um endpoint (resistir — KISS).

---

### M1 — [x] Modelo de skill + validação rígida (formato Theokit)

**Objective:** Implementar o modelo de domínio Skill/SkillRevision com o **formato `SKILL.md`
nativo do Theokit** e validação de payload rígida, mais o CRUD completo com revisões imutáveis.

**Definition of done:**

- [x] Parser de frontmatter `SKILL.md` compatível com o Theokit (`name/version/requires/description/allowed-tools/category/dependencies`); skill malformada é rejeitada na fronteira com erro tipado.
- [x] Validação de payload (zip): limites de itens/tamanho/ratio/profundidade, bloqueio de path traversal (`..`), symlinks e nomes duplicados; `SKILL.md` obrigatório na raiz. Inclui secret scan.
- [x] CRUD completo: `Create/Get/List(paginado)/Update(updateMask)/Delete` + revisões imutáveis; `skillId` validado (charset, imutável) e reservado pós-delete (janela configurável, não 24h fixos).

**Dependencies:** M0.

**Top risks:**

1. Divergência sutil do frontmatter Theokit quebra a integração de M7 (mitigar lendo `openskills`/`anthropic-skills`).
2. Validação de zip frouxa permite payload perigoso (zip bomb, traversal) — testar edge cases.

---

### M2 — [x] LRO robusto + governança + webhook

**Objective:** Tornar as operações de longa duração confiáveis e observáveis, com webhook de
conclusão além do polling — diferencial sobre o Google.

**Definition of done:**

- [ ] Operações idempotentes com estados explícitos (`CREATING/UPDATING/DELETING/ACTIVE/FAILED`), retry com backoff em falha transitória e nenhum retry em violação de regra de negócio.
- [ ] `GET /v1/operations/{id}` (polling) **e** entrega de webhook/evento na conclusão (assinatura por skill/projeto), com retry de entrega.
- [ ] Auditoria: toda mutação registrada de forma rastreável; teste de integração cobrindo sucesso, falha e reentrega de webhook.

**Dependencies:** M1.

**Top risks:**

1. Webhook sem retry/assinatura vira ponto cego de entrega — desenhar entrega ao menos "at-least-once".
2. Operações presas/inconsistentes sob falha do worker — timeouts e reconciliação.

---

### M3 — [x] Embeddings plugáveis + indexação (pgvector)

**Objective:** Gerar e indexar embeddings das skills para habilitar busca por intenção, com
provider de embeddings plugável (DIP), como nos demais serviços Theo.

**Definition of done:**

- [ ] `EmbeddingProvider` (port) com adapters `openai` / `local` / `stub` (mesmo trio da casa); teste unitário usa `stub`.
- [ ] Cada revisão gera embedding de `displayName + description + corpo SKILL.md`; coluna `vector` (pgvector) com índice; reindexação idempotente na atualização.
- [ ] Teste de integração: criar skill → embedding presente e consultável; troca de provider não toca o domínio.

**Dependencies:** M1.

**Top risks:**

1. Acoplamento ao modelo de embeddings (mitigado por DIP); dimensão do vetor fixada cedo demais.
2. Custo/latência de embedding no caminho da LRO — gerar de forma assíncrona no worker.

---

### M4 — [x] Busca híbrida transparente (RetrieveSkills)

**Objective:** Entregar a descoberta por intenção que supera o Google: híbrida (keyword +
vetor) com reranking e **score explícito** na resposta.

**Definition of done:**

- [ ] `GET /v1/skills:retrieve?query=...&topK=...` combina busca lexical (Postgres FTS) + vetorial (pgvector) com fusão/rerank e retorna `score` por resultado.
- [ ] Conjunto de avaliação interno: **Recall@5 ≥ 0.85** e retrieve **p95 < 200ms** medidos e reproduzíveis.
- [ ] Métrica de **time-to-relevant-skill** instrumentada (north-star) no caminho do retrieve.

**Dependencies:** M3.

**Top risks:**

1. Qualidade da busca abaixo do alvo — conjunto de avaliação desde o início; provider de embeddings trocável.
2. Fusão lexical+vetorial mal calibrada (pesos) degradando recall — tunar com o eval set.

---

### M5 — [ ] CLI de dev local (lint/validate/test)

**Objective:** Dar aos autores uma CLI que valida a skill localmente antes do upload —
capacidade que o Google não tem.

**Definition of done:**

- [ ] CLI valida estrutura `SKILL.md` + frontmatter Theokit + limites de payload + secret scan, com os **mesmos** checks da fronteira do servidor (regra única compartilhada — DRY).
- [ ] Comando empacota e publica a skill no registry (reusa a API de Create/Update); saída clara de erros por arquivo/regra.
- [ ] Teste E2E da CLI contra um registry local (validar → publicar → recuperar).

**Dependencies:** M1.

**Top risks:**

1. Divergência entre validação da CLI e do servidor (duplicação de lógica) — compartilhar o checker do `core`.
2. Secret scan com falsos positivos/negativos — usar padrões consagrados, não regex caseira frágil.

---

### M6 — [ ] RBAC granular por skill

**Objective:** Controle de acesso por skill (não all-or-nothing por projeto, como no Google),
com auditoria.

**Definition of done:**

- [ ] Modelo de permissões por skill (ler/escrever/publicar/deletar) aplicado em todos os endpoints, com negação clara (`403`) e sem vazar existência indevidamente.
- [ ] Autenticação na fronteira + audit log de acessos e mutações por principal.
- [ ] Testes de integração cobrindo concessão, negação e escalonamento de privilégio tentado.

**Dependencies:** M1, M2.

**Top risks:**

1. Modelo de RBAC complexo demais para V1 (over-engineering) — começar com papéis mínimos por skill.
2. Checagem de permissão inconsistente entre rotas — centralizar no middleware.

---

### M7 — [ ] Integração Theokit (remote skill provider) + dogfood

**Objective:** Fechar o V1: entregar o provider remoto que o Theokit consome e validar com
dogfood real — é o critério de "shipped".

**Definition of done:**

- [ ] `RemoteSkillsManager` (ou provider equivalente) para o Theokit: busca skills do registry via HTTP (`list` + `retrieve` semântico), com cache local e **fallback** para `.theokit/skills/` em falha do registry.
- [ ] Formato retornado casa com o `Skill { name, description, source, version, category? }` do Theokit; um agente Theokit real resolve `@Skills([...])` a partir do registry remoto.
- [ ] **Dogfood real** registrado: um agente Theokit interno usando o registry em uso de verdade; Recall@5 ≥ 0.85 e p95 < 200ms confirmados nesse uso.

**Dependencies:** M4.

**Top risks:**

1. Contrato com o Theokit muda durante a integração — alinhar a interface cedo com o time do Theokit.
2. Fallback mal feito deixa o agente sem skills em falha de rede — testar o caminho de degradação.

---

### M8 — [ ] Hardening + observabilidade por skill

**Objective:** Levar o V1 a qualidade de produção: observabilidade por skill, rate limiting,
SLO e cobertura E2E.

**Definition of done:**

- [ ] OpenTelemetry: traces + métricas por skill e por operação (incl. error budget e time-to-relevant-skill como métrica de primeira classe).
- [ ] Rate limiting por principal e SLO de retrieve documentado (p95 < 200ms) com alarme de regressão.
- [ ] Suíte E2E dos fluxos críticos verde no CI; documentação de operação publicada.

**Dependencies:** M2, M4, M7.

**Top risks:**

1. Observabilidade adicionada tarde demais para guiar tuning de M4 — instrumentar incrementalmente desde M4.
2. Rate limiting sem backpressure coerente — definir limites a partir de medições reais, não chute.

## Revision protocol

This roadmap is a living document but NOT a freeform scratchpad. To modify:

- **Marking progress:** flip the checkbox in the milestone header. No commit message ceremony required.
- **Adjusting a milestone's DoD:** edit in place, note the date in CHANGELOG.md.
- **Adding a milestone post-M8:** the project has outgrown its initial scope — write a new roadmap revision (e.g. `ROADMAP-v2.md`) rather than inflating this one.
- **Removing a milestone:** mark it `~~M3 — [-] name (cancelled YYYY-MM-DD — reason)~~` rather than deleting. The history matters.
